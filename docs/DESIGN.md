# SightMate 设计文档

## 用户故事

### 计划实现

1. 作为用户，我希望打开摄像头后，AI 可以理解当前画面。
2. 作为用户，我希望通过语音向 AI 提问。
3. 作为用户，我希望 AI 能用文字回答我。
4. 作为用户，我希望 AI 回答后可以自动语音播报。
5. 作为用户，我希望看到历史对话记录，包括问题、回答、截图和时间。
6. 作为用户，我希望可以清空历史记录。
7. 作为用户，我希望系统不要上传原始高帧率视频流，保护隐私并控制成本。
8. 作为开发者，我希望前端不直接请求阿里云 API，避免暴露 API Key。

### 最终实现

- 已实现：摄像头实时预览。
- 已实现：实时麦克风音频流输入，并保留浏览器语音识别回退。
- 已实现：视频对话模式，一个按钮同时开启摄像头、麦克风和实时模型连接。
- 已实现：实时模式把摄像头画面压缩为约 1fps JPEG 帧；回退模式截图最长边不超过 1024px。
- 已实现：前端通过 WebSocket 发送音频流和低帧率画面，回退时发送问题、截图和最近上下文到后端。
- 已实现：后端调用阿里云百炼视觉模型。
- 已实现：前端展示 AI 回答。
- 已实现：AI 回答后优先使用阿里云 Qwen-TTS 专业语音播报，失败时回退浏览器播报。
- 已实现：停止播报和重新播放。
- 已实现：SQLite 保存全部历史对话会话，每个会话包含多轮问答。
- 已实现：发送请求时只带最近 3 轮上下文。
- 已实现：清空历史记录。
- 已实现：loading、防重复提交、错误提示和隐私提示。
- 已实现：Qwen-Omni-Realtime 模式，前端通过后端代理发送麦克风 PCM 流和 1fps 摄像头帧，不暴露 API Key。

### 未完成的用户故事及原因

- 未实现：云端账号体系和跨设备历史同步。原因是当前使用本地 SQLite，避免引入云数据库和额外成本。
- 未完成：WebRTC 级全双工实时通话。当前使用 WebSocket 发送 PCM 音频和低帧率视频帧，已接收流式音频并播放；不是运营商级低延迟 WebRTC 通话。

## 成本控制策略

### 计划考虑的策略

1. 不上传原始高帧率视频流。
2. 实时模式使用低帧率抽帧；回退模式在用户主动提问时才截图。
3. 压缩图片后再上传。
4. 只保留最近 3 轮上下文用于模型请求。
5. 语音识别使用浏览器能力，语音播报优先使用专业 TTS。
6. 增加按钮 loading，避免重复调用。
7. 使用 SQLite 保存历史，避免引入云数据库。
8. 前端不直接请求模型服务，统一由后端管理 API Key 和模型调用。

### 实际采用的策略

1. **低帧率实时视觉**：实时模式只发送约 1fps 的压缩 JPEG 帧，避免原始视频流带来的持续带宽和推理成本。
2. **事件触发式锁帧回退**：实时模型不可用时，只有用户开始说话触发连续对话请求才锁定当前画面。
3. **图片压缩**：前端使用 canvas 将画面转成 JPEG base64，实时帧最长边约 480px，回退截图最长边限制为 1024px，减少模型输入体积。
4. **上下文裁剪**：SQLite 完整保存历史记录，但每次请求只发送当前会话最近 3 轮上下文，控制 token 成本。
5. **语音能力分层**：实时模式使用模型流式语音；回退模式语音识别使用 Web Speech API，语音播报优先使用 Qwen-TTS，失败时回退 `speechSynthesis`。
6. **本地历史存储**：使用 SQLite 保存会话历史，不部署云数据库。
7. **防重复提交**：模型请求 loading 期间禁用发送按钮，避免用户连续点击造成重复调用。
8. **后端代理模型请求**：所有阿里云模型请求经过 FastAPI 后端，前端不暴露 API Key，便于后续统一做限流、日志和错误处理。

### 为什么能降低成本

- 原始视频流通常意味着持续带宽和持续模型推理，低帧率抽帧能明显降低视觉输入成本。
- 图片越小，上传耗时和多模态模型输入成本越可控。
- 上下文越短，文本 token 成本越低，响应速度也更稳定。
- 语音识别不产生额外云服务调用费用；专业 TTS 只在回答生成后调用一次。
- SQLite 避免云数据库、对象存储和外部历史查询成本。
- 防重复提交可以减少误触导致的重复模型调用。

## 核心流程

```text
点击开始通话 -> 自动开启摄像头和麦克风 -> 前端通过 WebSocket 发送 16k PCM 音频和约 1fps 压缩画面 -> 后端代理连接 Qwen-Omni-Realtime -> 前端播放流式语音并展示字幕 -> 回答完成后保存到 SQLite 当前会话
```

实时模型不可用时，系统回退到按需锁帧流程：

```text
用户开始说话 -> 立即锁定当前帧 -> 识别到完整问题 -> 后端调用视觉模型 -> 后端合成专业语音 -> 前端展示并播报 -> 保存到 SQLite 当前会话
```

## 接口设计

### GET /health

返回：

```json
{
  "status": "ok"
}
```

### POST /api/vision/chat

请求：

```json
{
  "question": "请描述一下当前画面",
  "image_base64": "data:image/jpeg;base64,...",
  "history": [
    {
      "role": "user",
      "content": "画面里有什么？"
    },
    {
      "role": "assistant",
      "content": "画面里有一台电脑和一个水杯。"
    }
  ]
}
```

响应：

```json
{
  "answer": "当前画面中可以看到一台电脑、一个水杯和一些桌面物品。",
  "model": "qwen3.7-plus",
  "created_at": "2026-06-12 20:00:00"
}
```

### POST /api/speech/synthesize

请求：

```json
{
  "text": "你好，我是 SightMate。"
}
```

### GET /api/sessions

返回 SQLite 中保存的本地会话列表。

### POST /api/sessions

创建一个新的本地会话。

### POST /api/sessions/{session_id}/turns

向指定会话追加一轮问答。

### DELETE /api/sessions/{session_id}

删除指定会话。

### DELETE /api/sessions

清空所有本地会话。

### WebSocket /api/realtime/ws

后端 Qwen-Omni-Realtime 代理。前端连接该地址后，由后端携带 `ALIYUN_API_KEY` 连接阿里云实时模型，避免浏览器暴露密钥。前端会发送：

- `input_audio_buffer.append`：16k PCM base64 麦克风音频。
- `input_image_buffer.append`：约 1fps 的 JPEG base64 摄像头帧。

前端会接收并播放 `response.audio.delta`，同时展示 `response.audio_transcript.delta/done` 形成实时字幕和历史记录。

响应：

```json
{
  "audio_url": "https://...",
  "format": "mp3",
  "model": "qwen3-tts-flash"
}
```

## 隐私设计

- 摄像头实时预览只在浏览器本地展示。
- 实时模式不上传原始高帧率视频流，只发送约 1fps 的压缩 JPEG 帧。
- 回退模式在用户开始说话时锁定当前帧，并只在生成回答时上传这一张截图。
- 前端不展示截图预览，避免让用户误解为手动上传图片流程。
- 真实 API Key 只保存在后端 `.env` 中。
- 历史记录默认保存在后端本地 SQLite 文件中。
